# Design Artifacts

当用户要求生成 HTML 页面、原型、报告、图表或 artifact 时，用这份指南判断设计方向。

目标不是套模板，而是在 Anthropic / Claude 的气质范围内做清楚、有判断力、可使用的页面：温暖、克制、排版强、信息密度合理。

## 0. 工作方式

动手前先判断：

1. 这页真正要帮用户完成什么？
2. 用户第一眼最需要看懂什么？
3. 哪些事实、数据、引用、文件名、状态或交互结果必须保真？

如果目标不清楚，先问用户。  
如果目标清楚但用户指定的实现路径很绕，先说明更短路径；在不改变用户明确约束和交付形式（如单文件、不用 JS、需复刻某结构）的前提下可采用，若会改变用户指定方案则先确认。

你可以自由决定页面结构、字体比例、图形形式、组件组合和交互复杂度。不要擅自改变用户给出的事实、结论、品牌名、日期、价格，以及法律、医学、金融等高风险内容。

## 1. 风格内核

接近 Anthropic / Claude 的科技艺术哲学并存气质：

- 温暖纸张感，而不是冷科技 dashboard。
- 强排版、清楚层级、克制留白，而不是堆装饰；留白服务于阅读节奏和分组，而不是制造空旷感。
- 细线、表格化信息、明确分组，让用户容易扫描和判断。
- 标题可以有表达性，正文必须清楚、具体、可信；文案冷静、可判断，避免空泛营销词。
- 图像、插画、图表和交互都服务内容，不为了填空。
- 克制动效：状态变化可以轻微、有意义，不做炫技。

不要按页面类型（“产品页”“新闻页”“报告页”）套固定结构；结构由内容关系决定（方法见第 3 节）。

可见文案必须面向最终读者和内容本身，不面向用户、Agent 或设计过程。不要解释设计过程、视觉手法、布局选择、颜色含义或装饰性隐喻（设计意图体现在画面和结构里，不写成说明文字）；但允许并鼓励为图表、数据、控件和复杂交互提供面向读者的 caption、legend、使用提示和状态说明。

风格判断：

- 默认按报告型 / 学术期刊型来组织：首屏即承担信息，标题尺寸克制，立刻进入正文、数据、目录或摘要（首屏密度细则见第 3 节，营销页式首屏的禁止见第 8 节）。
- 优先做成研究机构、学术期刊和公共利益组织的混合气质，而不是 SaaS 营销落地页、产品发布页或聊天产品后台。
- 像经过编辑的研究出版物：标题强、正文安静、分隔克制、信息关系清楚；靠排版和细线建立层级，不靠大量卡片、阴影、渐变或产品 mockup 制造设计感。
- 信息密度中等偏高,接近期刊和年度报告的版面感——一屏能看到多层信息(标题 + 摘要 + 数据 + 列表/表格/图),但正文和分组之间保留可呼吸的节奏,不要密到像 dashboard 或后台表格。
- cream canvas、editorial typography、warm accent、few shadows、soft rounded controls 是可取方向。
- 如果不确定该偏产品感还是出版物感,优先选择出版物感:更少装饰、更轻边线、更强排版、更明确的信息层级。

## 2. 基础视觉

推荐色彩：

- 背景：`#f7f4ea`、`#faf9f5`、`#ffffff`
- 文字：`#141413`、`#3d3d3a`、`#6f6e68`
- 线条：`#d1cfc5`、`#d8d1c3`、`#e8e6dc`、`#e9e4d8`（最常用 `#d8d1c3`、`#e9e4d8`）
- 低饱和辅助色：`#e7ded0`、`#c1d6cf`、`#788c5d`、`#b04a3f`。其中 `#b04a3f`(rust)与橙系相近，只用于插画色块，不在 UI 里与 Claude 橙争夺焦点。
- Claude 经典橙：`#d97757` 是唯一的重点色 / 强调色，用于品牌暖色、插画焦点、图标填充，以及当前项、选中态、短 underline 或单个重点；深一档可用 `#c46849`。它只点缀少量焦点，不替代排版和细线本身承担的信息层级（铺大面积的禁止见第 8 节）。
- 橙更适合图标、underline、色块焦点和大号数字；用作正文级或小号文字 / 关键数据时对比度可能不足，此时改用深文字色，仅用橙做旁注标记。

线条、圆角和层级：

- UI 线条默认使用上面的暖灰细线。黑色粗线属于插画语言，不要搬到按钮、卡片、输入框或 tab 上（见第 8 节）。
- 信息分隔优先用 `0.5px-1px` 横线、留白和排版节奏；只有真正需要框定、比较、选择或输入的元素才加完整边框。
- 圆角按层级使用：列表、正文分隔、ruled rows 通常 `0px`；按钮和轻量标签 `8px`（主、次级按钮同档，次级不要比主按钮更圆）；输入框、tabs、segmented control `10-12px`；图片和媒体容器 `16px`；大产品卡、模型卡、重点面板 `24-32px`。
- `999px` 胶囊只用于少量 filter chip、状态 pill 或非常小的 metadata，不要让所有 metadata、按钮和标签都胶囊化。
- 默认不要用投影制造层级。除 modal、popover、浮层菜单外，避免 `box-shadow`；hover 也优先改变 underline、边线深浅、背景明度或箭头位置。

字体：

- sans 用于正文、界面、按钮、表格、metadata。
- serif 可用于大标题、引言或少量叙事，增加温度。
- mono 用于代码、文件名、技术输出、数据片段。
- 数字偏学术、严谨、安静，像研究报告里的关键数值而非广告海报的视觉噱头：优先用正文 sans / serif 的 lining + tabular numerals（`font-variant-numeric: lining-nums tabular-nums;`、`font-feature-settings: "lnum" 1, "tnum" 1;`），让日期、价格、统计和表格的位数对齐。
- 避免 display、压缩、几何赛博、复古数码管 / 仪表盘、描边或渐变等装饰感数字。
- 只有代码、日志、哈希、命令行输出才用 mono 数字；普通统计和学术比较不要因为“是数字”就切到等宽。
- 大标题只给首屏或章节转折；工具面板、卡片、表格内标题要更小、更密。

## 3. 布局原则

优先从信息关系出发，而不是从页面类型出发。动手前先判断这份内容是哪种关系，再从下面选最贴合的结构；同一份内容可能命中多种，以“读者最需要先看懂什么”为准。

内容关系 → 结构：

- 一段主线叙述 → 大标题 + 简短说明 + 下方主体内容。
- 说明与主体并置 → 两栏：一侧标题或说明，另一侧正文、列表、表格、图形或操作区。
- 逐条罗列（日期、类别、标题、链接、资源、联系人、轻量比较）→ ruled rows。
- 横向比较（方案、作品、模型、价格、案例）→ grid。
- 沉浸阅读长文（分析、说明、代码）→ 单主栏。
- 强导航长内容（长文档、源码解读、多章节报告、论文精读）→ sticky 侧边目录 + 主栏：`992px+` 可用左侧固定目录（`position:sticky`）配合滚动高亮当前章节；不强制，由模型按内容长度和章节数自行判断，短页、单章或营销页不需要。
- 操作并看结果（计算器、模拟器、筛选器、prompt 原型、可调参数工具）→ 控件 + 结果。

布局约束：

- mobile 左右 padding 通常 `20-24px`。
- desktop 左右 padding 通常 `48-80px`。
- grid gap 通常 `24-40px`。
- 正文不要铺满超宽屏；把额外空间给图像、side rail 或留白。
- 固定格式元素要有稳定尺寸、比例或网格，避免 hover、标签、加载文字导致布局跳动。

首屏密度：

- 首屏 H1 通常控制在 `40-64px`（desktop），不要做到 `96px+` 的 hero 字号；标题旁或下方必须立刻承载实质内容：摘要段、关键数据、目录、ruled list、章节导航或主图 + caption。
- 首屏可见区域应让读者看到至少 2-3 层信息（如标题 + 摘要 + 数据组，或标题 + 目录 + 一段引文），而不是只看到一个标语。
- 营销落地页式首屏默认禁用，只有用户明确要求做 landing page、发布页、营销页或封面页时才允许（形态与禁止见第 8 节）。

## 4. 隐喻插画

HTML 首页或核心页默认必须包含一张偏艺术风格的隐喻插画，除非用户明确要求不包含。它通过形体、材质、关系和留白建立**主题气质和核心隐喻**，不负责呈现流程、架构、数据或界面说明。

硬性约束：

- 隐喻插画内部不允许出现任何中文、英文或数字形式的文字信息，包括标题、单字、缩写、编号、日期、刻度、标签、按钮文案、水印、代码、伪文字线条或乱码字形。
- 需要传达精确文字或数字时，放在插画之外的 HTML 标题、正文、caption、列表、表格或独立解释型图形中。
- 不要用流程框、坐标轴、产品界面、代码片段或系统架构图代替隐喻插画。

画面语言：

- 一般情况下使用透明背景，让插画直接融入页面底色；只有内容明确需要独立画面底面时，才引入局部或完整背景。
- 黑色手绘粗线、松弛曲线、圈注形状或下划线形态。
- 抽象的人、手、脸部轮廓、器物、纸张、窗口、碎片或符号性形体。
- 低饱和色块：sage（`#c1d6cf`）、sand（`#e7ded0`）、rust（`#b04a3f`）、olive（`#788c5d`），配少量白色或黑色图形。
- 纸张、网格、剪贴感、轻微错位、手工痕迹。
- 一两个核心形体、大面积留白，以及至多一处重点色（Claude 橙）。

使用原则：

- 先找到与内容有关的一个核心隐喻，再画画面；不要先画装饰再为它补解释。
- 插画与正文中的解释型 SVG / 图表必须明显区分，不能把节点、矩形和箭头组成的系统图挪到首屏充当插画。
- 不描述形状，也不解释插画隐喻。

避免（通用视觉禁忌见第 8 节）：

- emoji 式图标、SaaS 人物插画、isometric 3D。
- 标签堆砌、复杂连线蜘蛛网，或看起来很忙但不增加理解的装饰。

## 5. SVG 图形

SVG 是图形实现工具，不等于插画。HTML / CSS 负责排版、纸张质感、分组和响应式；SVG 负责隐喻插画、流程图、状态图、封面小图、缩略图和内容型图表。用 SVG 实现隐喻插画时，同样严格遵守第 4 节的无文字约束。

- 优先使用 inline SVG，便于响应式、主题色、hover / selected 状态和小幅动效。
- 隐喻插画多用 `path`、不规则形体、粗线、留白和少量色块（无文字约束遵循第 4 节）。
- 解释型图形可用 `rect`、`line`、`circle`、`polyline`、`text`，保持少量节点和清晰标签。
- 一张解释型图围绕一个主叙事；可含必要的次级变量和注释，但不要让读者同时解读多个无关结论。
- 可以用 CSS 变量控制 `stroke`、`fill`、`opacity`、`transform`。
- 复杂图表或重复图形可以由 JS 生成 SVG，但输出应保持可读、可维护。
- SVG 应有稳定 `viewBox` 和容器比例，移动端不能挤压变形。

## 6. 组件和交互

组件选择：

- 卡片只用于需要边界来比较、选择或点击的重复对象。
- 普通正文 section 不要包成浮动卡片。
- 新闻、研究、报告、资源列表优先做 ruled layout：标题、metadata、摘要和链接用横线与网格组织，不要把每条内容自动做成小卡片。
- 主按钮可以黑底白字，高度通常 `36-44px`；次级按钮用白底或透明底 + 浅暖灰边框（圆角层级见第 2 节）。当某个行动点确实是页面主 CTA 时，主按钮也可用 Claude 橙底白字，但同屏只用一次。
- 小按钮、链接按钮、工具按钮要有稳定高度和内边距，不要做成尖锐小矩形；图标按钮保持正方形触控区。
- 搜索框和输入框通常高 `40-44px`，白底或近白底，`1px` 浅边框；placeholder 和图标用 muted 灰，不要高对比抢正文。
- tabs / segmented control 可用浅米色底或无框容器；active 项用白底或轻背景和清楚文字权重，inactive 项保持透明或低对比。
- filter、tabs、segmented control、slider、toggle 都要有可见状态。
- metadata 不要全部做成胶囊；新闻日期、类别、来源更适合小号文字、细分隔线或 inline metadata。

交互原则：

- 交互要改变理解、选择或状态，不要用交互遮盖信息结构，也不要把关键信息只放在 hover 里（见第 8 节）。
- 搜索、筛选、排序、tabs、segmented control、accordion、row selection、slider、toggle 都可以自由使用。
- 按控件角色同步正确语义状态：tabs / listbox 用 `aria-selected`，toggle 按钮用 `aria-pressed`，checkbox / radio / switch 用 `checked` 或 `aria-checked`；可见文本只作补充，不能替代语义状态。
- 控件改变图形时，用 `data-state`、`data-step`、`data-mode` 或 CSS variables 驱动视觉。
- dropdown / popover 支持 `Esc` 和点击外部关闭。
- `prefers-reduced-motion: reduce` 下禁用循环动画、路径绘制和自动 staged reveal。

动效建议：

- hover：underline、边框略深、背景轻微变暖、箭头轻移、卡片上移 `1px`。避免 hover 时突然出现明显阴影或大面积高饱和强调色底。
- 展开/收起：`150-220ms`，只改变 opacity、translate 或 height。
- 图形状态：快速切换 fill、stroke、opacity、transform。

## 7. 响应式和实现

以 web 端大屏效果为主要目标；移动端至少保证可读、可操作、不横向溢出，复杂排版可简化，不要求与桌面端同等精细。下面各断点的退化方式必须落实。

- `<640px`：单列；H1 降低；rows 堆叠；复杂表格可横向滚动。
- `640-991px`：主内容单列；grid 可 2 列。
- `992px+`：可使用两列 hero、featured grid、side rail。
- `1280px+`：不继续放宽正文。

实现要求：

- 使用语义化 HTML：`header`、`main`、`section`、`article`、`nav`、`footer`。
- CSS 放在页面内或项目约定位置；避免依赖外部 CDN，除非用户允许。
- JS 只承担必要交互，不要让静态页面复杂化。
- 真实数据优先；没有数据时，用少量可信 placeholder，并让结构清楚。
- 图片如果没有来源，不要硬塞 stock 感图片；用信息型 SVG、代码块、表格或纯排版替代。
- 所有交互控件都要能键盘访问。

## 8. 硬性禁止

分两类。第一类是风格禁忌，用户明确要求时可放开；第二类是可用性底线，即使用户要求也不放开。

风格禁忌（用户明确要求时可放开）：

- 营销落地页式首屏：超大 hero 字、居中大标语、大面积空白、首屏只有一个 CTA、产品截图占满首屏。
- 紫蓝大渐变、玻璃拟态、发光边框。
- 大面积高饱和强调色背景（包括把 Claude 橙铺成大块底色）。
- 每个 section 都包成浮动卡片。
- 把每个小组件都做成尖锐矩形，或用黑色粗边框强调普通 UI 控件。
- 默认给卡片、按钮、表格、图片加明显投影。
- 纯装饰 SVG、generic blob、发光球、抽象弥散背景。
- 图标堆砌来替代真实信息。
- 过度圆润、可爱、移动 App 化的 UI。
- 首屏大量元素飞入、持续 pulse、parallax。

可用性底线（即使用户要求也不放开）：

- hover 才能看到关键信息。
- scroll-jacking 劫持滚动。
- 移动端暴露未样式化 nav、表格、列表或控件，或窄屏破版、横向溢出。
- 交互控件无法键盘访问。

## 9. Notes
1. 当规则冲突时，优先级是：用户目标和真实内容 > 可读性、可访问性、响应式 > 信息结构和判断效率 > Anthropic / Claude 气质 > 具体色值、圆角、字号、组件建议。
